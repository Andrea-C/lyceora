# Lyceora M2.1 — Roles, Navigation, Admin Dashboard, UX Fixes

**Date:** 2026-07-16 · **Status:** approved by Andrea (roles: Better Auth admin plugin; impersonation: full act-as; dashboard: users+stages+drill-in) · **Origin:** `docs/2026-07-15-improve-Milestone-2.md` (PM usability request) · **State:** M2 live (tag `m2`), tests 153/153.

## Context

Andrea, testing as PM, hit: silent birth-year rejection, no signal that a created profile must then be clicked, no navigation (parent area reachable only by URL, no logout), and no way to see all users/profiles or experience the app as another user. Decision taken after discussion: real-but-minimal roles now (the prototype is public production holding a minor's data — the cross-tenant dashboard must be gated from day one), full impersonation for PM testing, all tenant-scoped features visible to every account.

## 1. Roles & impersonation

- Enable Better Auth **admin plugin** (`better-auth/plugins` `admin()`) in `apps/web/src/lib/auth.ts` and the client plugin (`adminClient()`) wherever the auth client is built.
- Schema: add the plugin's fields to `packages/db/src/auth-schema.ts` — `user.role text default 'parent'`, `user.banned boolean`, `user.banReason text`, `user.banExpires timestamp`, `session.impersonatedBy text` (exact field set per the plugin's schema reference; generate migration 0005). Roles used: `admin`, `parent` (plugin default role configured to `parent`).
- Seed: one-off SQL at deploy — `UPDATE "user" SET role='admin' WHERE email='andrea.clementi.ac@gmail.com';` (documented in README deploy notes; also run against local dev DB).
- `requireAdmin()` helper in `apps/web/src/server/http.ts` family: resolves the session, returns the user when `role === 'admin'`, otherwise 404s (`notFound()`) for pages / 403 JSON for routes. EVERY admin page/action/route uses it — defense in depth on top of plugin-side checks.
- **Impersonation:** dashboard row button calls the plugin's impersonate endpoint (client `authClient.admin.impersonateUser({ userId })`), then redirects to `/{locale}/app/profiles`. While `session.impersonatedBy` is set, a fixed banner (`ImpersonationBanner`, client component) shows "Stai impersonando {email}" + "Torna admin" which calls `stopImpersonating`, **clears the `lyceora_profile` cookie**, and redirects to the dashboard. Plugin default impersonation session duration (1h) is acceptable.
- Non-goals: `student` login role (children remain credential-less profiles; M3 topic), granular permissions, ban/delete UI.

## 2. Navigation (`AppNav`)

Server component rendered from `apps/web/src/app/[locale]/app/layout.tsx` (create the layout if the segment lacks one) on every `/app` page:

- Left: "Lyceora" wordmark → `/{locale}/app`.
- Items: Home (`/app`), Sessione (`/app/session`) — both only when a profile cookie is set and valid; Area genitori (`/app/parent`); Dashboard (`/app/admin`) admin-only; active-child chip (displayName) → `/app/profiles`; Esci (server action: Better Auth sign-out + clear `lyceora_profile` cookie + redirect `/{locale}`); existing EN/IT toggle moves into the bar (top-right).
- Kid-appropriate: plain text links, existing zinc/rounded idiom, wraps on mobile (no hamburger).
- i18n namespace `nav`: `home`, `session`, `parent`, `dashboard`, `switchProfile` (used as chip title), `logout` — it + en.
- The diagnostic and session pages keep their focused layouts (nav shows, content unchanged).

## 3. Admin dashboard

- `/{locale}/app/admin` (page) + `/{locale}/app/admin/profile/[id]` (drill-in), both `requireAdmin()`-gated server components.
- **Counters row:** total users, total profiles, learning sessions started in last 7 days, profiles active in last 7 days (distinct profiles with `daily_activity` rows).
- **Users table:** one section per user — email, role chip, created date, Impersonate button — with that user's profiles listed inline: displayName, stage badge (`diagnostica da fare` when no completed diagnostic session, else `in percorso`), mastered-topics count, total XP, current streak, last active date.
- **Drill-in:** renders the shared `ProfileReport` for any profile id (no ownership check — admin gate replaces it; the component itself takes data, not authority).
- **Refactor for reuse (targeted, addresses an M2-review finding):** extract the parent page's per-child report sections (domain bars incl. bucketing, `ActivityChart` + 14-day zero-fill, weekly summary, "Da rivedere insieme", recent badges) into `apps/web/src/components/ProfileReport.tsx` (server component taking `{ profileId, locale }`, doing its own data fetches via a shared server helper `apps/web/src/server/services/profile-report.ts`). Parent page and admin drill-in both consume it. Home page's domain bucketing may also use the shared helper if trivial; do not force it.
- i18n namespace `admin`: `title`, `users`, `impersonate`, `stagePending`, `stageActive`, counters labels — it + en.

## 4. UX fixes

- Birth-year label: "Anno di nascita (2005–2022, facoltativo)" / "Birth year (2005–2022, optional)" — replaces the bare label; input keeps `min`/`max`.
- **Auto-enter on create:** `createProfile` action sets the `lyceora_profile` cookie to the new profile id and `redirect(`/${locale}/app`)` (mirrors `selectProfile`). The card list remains for switching. Update the E2E, which currently clicks the card after creating.
- Logout + parent link: delivered by §2.

## 5. Security & testing

- Any non-admin hitting `/app/admin*` gets 404 (not 403 — don't advertise the route). Impersonation endpoints are plugin-role-guarded; our UI only exposes them to admins.
- Tenant-scoped routes/actions keep their existing `getOwnedProfile` gates — impersonation works BECAUSE the session becomes the target user, not by bypassing gates.
- Tests: PGlite integration — `requireAdmin` (parent blocked, admin passes); dashboard data assembly (stage badge logic, counters); `ProfileReport` helper parity (parent page data unchanged after refactor). Component — AppNav renders per role/context (admin sees Dashboard, parent doesn't; chip when profile selected). E2E — updated create-profile auto-enter flow; nav navigation to parent area; logout returns to landing.
- Rollout: `feature/m2.1`, per-task implementer+reviewer process, final review, merge → auto-deploy, prod migration 0005, admin seed SQL, prod smoke (nav visible, dashboard 404 for a fresh non-admin account, dashboard renders for Andrea after seed).

## Acceptance criteria

All tests green (153 baseline + new); nav bar on every app page with role-correct items; creating a profile lands the user in the app; a non-admin account gets 404 on `/app/admin`; Andrea's account sees the dashboard with counters, all users, profile stages, and working drill-ins; impersonation round-trip works (banner shown, revert restores admin, profile cookie cleared); parent page renders identically to pre-refactor (component reuse, no visual change); deployed and smoke-checked on app.lyceora.ai in both locales.
